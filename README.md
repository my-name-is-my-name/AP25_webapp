# AP-25 RAG

RAG-система для работы с АП-25: точечных нормативных ответов и подбора пунктов для инженерного анализа.

Проект ориентирован не на "общий чат по авиации", а на более узкую задачу: помочь быстро найти нужные пункты АП-25, показать связанный контекст и снизить количество ручного поиска по документу.

## Что умеет система

Сейчас в проекте есть два основных режима:

- `normative_lookup`  
  Для прямых вопросов вида:  
  `Какой коэффициент безопасности применять по умолчанию?`  
  `Какие требования к независимости подачи топлива в двигатели?`

- `issue_spotting`  
  Для более широких инженерных вопросов вида:  
  `Какие пункты нужно проанализировать при изменении компоновки пассажирского салона?`  
  `Какие пункты АП нужно проанализировать при ремонте коррозионного повреждения?`

## Как устроена система

Логика проекта сейчас выглядит так:

1. PDF АП-25 разбирается на структурные фрагменты.
2. Для каждого фрагмента сохраняются:
   - основной текст
   - поисковое представление
   - заголовки и контекст
   - маркеры подпунктов
   - внутренние ссылки
3. Поиск идет гибридно:
   - по словам
   - по смысловой близости
4. Затем результаты ранжируются и собираются в короткий shortlist.
5. Дальше система либо:
   - отвечает на прямой нормативный вопрос
   - либо возвращает shortlist пунктов для последующей ручной проверки

Важно: проект сознательно уходит от слишком сложной post-hoc логики. Если уверенность низкая, система должна скорее показать shortlist и не додумывать лишнее.

## Что уже работает хорошо

Лучше всего система сейчас работает на прямых нормативных вопросах, где:

- вопрос короткий и конкретный
- в АП-25 есть один достаточно явный основной пункт
- ответ можно опереть на один paragraph family

Примеры хороших кейсов:

- `Какой коэффициент безопасности применять по умолчанию?`
- `Какие требования к независимости подачи топлива в двигатели?`
- `На какие нагрузки оценивать прочность конструкции при аварийной посадке?`
- `Какой дополнительный коэффициент безопасности для стыковочных узлов?`

## Где система пока слаба

Основные слабые места сейчас такие:

### 1. Не все простые вопросы стабильно выбирают главный пункт

Например, вопрос про расчетные перегрузки для пассажирских кресел уже находит нужный `§25.785(f)`, но не всегда делает его главным источником ответа.

### 2. Общие инженерные вопросы пока шумят

Например:

- `Какие пункты нужно проанализировать при оценке изменения пассажировместимости?`
- `Какие пункты АП нужно проанализировать при ремонте коррозионного повреждения ... ?`

Здесь retrieval уже иногда находит правильное ядро, но shortlist еще может разрастаться вторичными или слишком широкими пунктами.

### 3. Issue spotting еще зависит от качества shortlist

Если начальный retrieval слабый, то даже аккуратная post-processing логика не делает результат по-настоящему сильным. Поэтому следующий реальный шаг здесь — не добавлять новую "умную" логику, а улучшать retrieval и ranking.

## Почему в проекте нужна ручная разметка

Сейчас качество можно оценивать по нескольким показательным примерам, но для устойчивого улучшения этого уже недостаточно.

Нужна ручная разметка:

- типа вопроса
- правильных `§` и подпунктов
- обязательных и лишних пунктов для shortlist

Эта разметка нужна не "для красоты", а для двух практических задач:

1. Честно сравнивать версии системы.
2. Обучить более аккуратные routing и reranking-модели на реальных примерах.

## Как запускать

На практике основной рабочий вход сейчас — через notebook:

- открыть `AP25_parser_workbench.ipynb`
- выполнить setup-ячейки
- и использовать `ask_ap25(...)` для тестовых запросов

Дополнительно можно запускать простые регрессионные тесты:

```bash
python3 -m unittest -q tests/test_regressions.py
```

Установка зависимостей:

```bash
python3 -m pip install -r requirements.txt
```

## Браузерное приложение

В проект добавлен простой браузерный интерфейс, который можно поднять на удаленном сервере и открыть с других компьютеров в сети.

Запуск:

```bash
python3 ap25_webapp.py --host 0.0.0.0 --port 8000
```

Если хотите, чтобы retrieval и базовая модель прогрузились сразу при старте:

```bash
python3 ap25_webapp.py --host 0.0.0.0 --port 8000 --preload-retrieval --preload-default-model
```

После запуска приложение будет доступно по адресу вида:

```text
http://SERVER_IP:8000
```

Что уже есть в веб-версии:

- строка ввода в стиле простого chat UI
- выбор режима `auto / normative_lookup / issue_spotting`
- просмотр коротких retrieval-фрагментов под ответом
- выбор модели для тестов
- кнопка подключения локальной модели
- кнопка скачивания и подключения модели, если ее еще нет на сервере

## Поддержка нескольких LLM

Сейчас в веб-приложении заведены такие варианты:

- `Qwen2.5-1.5B-Instruct`  
  Базовый быстрый вариант для повседневных тестов

- `Qwen2.5-7B-Instruct`  
  Добавлен как рабочая интерпретация запроса про `qwen-7.5b`

- `Qwen3.6-35B-A3B`  
  Добавлен для более тяжелых сравнительных тестов

Важно:

- `Qwen3.6-35B-A3B` — тяжелая модель, для нее нужна действительно мощная GPU-инфраструктура
- на CPU или слабой видеокарте такая модель может быть непрактична
- для `Qwen3.6-35B-A3B` автоскачивание в коде отключено
- предполагается, что эта модель уже лежит локально на сервере
- при необходимости путь можно передать через переменную окружения `AP25_QWEN36_35B_A3B_PATH`

Пример:

```bash
export AP25_QWEN36_35B_A3B_PATH=/path/to/local/Qwen3.6-35B-A3B
python3 ap25_webapp.py --host 10.99.66.212 --port 8000
```

## Ближайшие шаги

Самые полезные следующие улучшения сейчас такие:

1. Сделать `normative_lookup` более стабильным на простых factual-вопросах.
2. Перейти к принципу `best-hit first` для выбора главного пункта ответа.
3. Сократить шум и повторы в `issue_spotting`.
4. Собрать evaluation set и начать измерять прогресс на размеченных примерах.
5. После этого — обучать более аккуратные routing и reranking-компоненты.

## Текущее состояние проекта

Это уже не "черновой демо-ноутбук", а рабочий исследовательский прототип с хорошим парсером, нормальной retrieval-архитектурой и понятной дорожной картой.

При этом проект пока еще не стоит воспринимать как полностью автоматический экспертный инструмент. В текущем виде он уже полезен как внутренний assistant для инженера, но ключевые решения по сложным кейсам по-прежнему требуют ручной проверки.
